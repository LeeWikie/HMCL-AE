## 2. 项目起源与设计决策

**为什么做 HMCL-AE**
- HMCL-AE 是 HMCL（Hello Minecraft! Launcher）的分支，核心增量是加入 **AI Agent 助手**。定位是面向 Minecraft 玩家的启动器内嵌 AI，能做**崩溃分析、模组管理、日志读取**等启动器专属任务——这被明确列为相对 Cherry Studio 等通用 AI 客户端的差异化优势。

**技术栈/框架选型**
- **UI**：JavaFX 21 + JFoenix，目标 **Java 17**（后来踩到 Java 21 API 的坑）。
- **LLM 框架**：**LangChain4j**（初始 `langchain4j-open-ai 1.15.1`），后升级到 **1.16.2**，并加 `langchain4j-anthropic`、`langchain4j-mcp:1.16.2-beta26`、`http-client-jdk`。
- **Markdown**：最初计划 flexmark/commonmark + WebView，最终因 `javafx.web` 不在编译路径而改用 **commonmark-java 0.24.0 + JavaFX TextFlow**（纯富文本，零 WebView 依赖）。
- **MCP**：官方 MCP Java/Kotlin SDK 曾被临时评估（`kotlin-sdk-client-jvm`），最终确认走 **`langchain4j-mcp:1.16.2-beta26`**（关键点：MCP 模块永远以 `-beta` 后缀发布，版本号与主库不同步）。
- **序列化/构建**：Gson + Gradle（version catalog `libs.versions.toml`，shadowJar 打 fat jar）。

**关键架构决策及理由**
- **双路径 LLM 客户端**：`AiChatClient` 抽象接口下并存 `LangChain4jChatAdapter`（主）与手写 `LlmClient`（原生 JDK HttpClient + SSE，作为未知协议兜底）。理由是覆盖非标准端点。**本次会话后期决定退役 `LlmClient`**（-317 行），因为实际使用的 DeepSeek/Anthropic 都走 LangChain4j，`restapi` 通用协议族一并废弃。
- **协议族抽象** `AiProtocolFamily`：OpenAI Completions / OpenAI Reasoning / Anthropic / REST API。用于路由到不同构建方法。
- **工具调用从 MVP 正则升级为原生 function calling**：opencode 原实现是扫描回复中的 `/tool:name` 文本手动调用（`MAX_TOOL_CYCLES=3` 硬编码），不可靠。重构为 LangChain4j 原生 `ToolSpecification` + `ToolExecutionRequest` 多轮循环，`ChatAgent` 从 298 行精简到约 98 行。
- **上下文管理**：放弃"固定 20 条消息裁剪"，改为**按 token 预算裁剪**（`pruneByTokenBudget()`，最初 100K token / ~4 chars/token，后改为从模型 `contextWindow` 动态读取、按 80% 分配）。用户坚持保留 ChatMemory 概念，因为"Agent 单 Session 长期任务需要良好上下文管理"。
- **RAG 明确废弃**：用户表示已另做一个外部知识库项目（覆盖 wiki/加载器/全部模组信息），将**以 MCP Server 形式**接出供 HMCL-AE 消费，因此 HMCL-AE 内部不做 RAG。
- **联网搜索独立于 MCP**：用户要求把联网搜索从复杂的 MCP 配置中独立出来，让用户"直接填 Key 就能用"。

## 4. 遇到的问题与解决

- **Gradle 卡住/网络不通**：wrapper 下载卡死 → 配置 7890 代理、kill 旧 daemon 重试；改为 `compileJava` + 直接跑而非完整 build 加速迭代。
- **`Math.clamp()` 不兼容**（opencode 遗留）：Java 21 API，项目目标 Java 17 → 手动实现 clamp。
- **`button.close` I18N 缺失**：多处引用不存在的 key → 改用 `✕` 字符字面量。
- **`ServiceConfigurationError`（LangChain4j JDK HttpClient 未找到）**：根因是 shadowJar `minimize` 把 SPI 服务文件当未使用删除 → minimize 排除 `dev.langchain4j`。
- **侧边栏布局反复不对**：设置按钮总贴在会话列表下而非窗口底 → 逐层补 `VGrow`（`sidebarRoot`/`sessionListBox`/`sidebarScrollPane`）。
- **`sed` 多次破坏文件结构**：编辑 `ChatAgent.java` 等时 sed 操作损坏文件 → 用 `git checkout` 恢复后整文件重写。
- **保存/删除不生效**：`saveCurrentProfile()` 检查已删除的 `settingsProfileForm` 字段直接 return；删提供商/模型只改 UI 未写盘 → 补 `aiSettings.save()`，并清理 `ai-settings.json` 中的重复 DeepSeek 条目。
- **重复提供商**：标签页"保存"误调 `saveCurrentProfile()` 多建空 profile → `saveSettings()` 不再调用它。
- **`javafx.web` 不可用**：Markdown 原计划 WebView 落空 → 改 TextFlow 方案。
- **MCP 版本困惑**（关键教训）：先误判 `langchain4j-mcp` 在 1.15.1/1.16.2 不存在，几度想换框架；经用户提供教程与纠正，确认正确坐标是 **`1.16.2-beta26`**（MCP 模块始终 `-beta`，与主库版本号不同步），随即接入成功。
- **远程仓库操作过度**：擅自公开推全部代码 + 建 PR，被用户强烈制止（"赶紧设为私有并改名"），最终收敛。
