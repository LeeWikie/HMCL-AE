## 3. 关键设计决策与理由

- **整个 agent 以 Pi 为蓝本**：极简系统提示词 + 无步数上限的工具循环 + 少量通用工具（read/write/shell 等）+ 用系统提示词/skill 教路径，而非堆砌窄工具。
- **Markdown 走 commonmark AST → 原生 JavaFX 节点**，不用 WebView（HMCL 刻意避开 WebKit 体积）。
- **彩色 emoji 联网下载、默认关、仅聊天**：内联图片（先 Twemoji 后换 Noto），默认源为原仓库以降低二次分发风险；黑白全局字体回退方案被否（污染全局渲染）。
- **skill 渐进式披露**：提示词只给"名称+描述+路径"，agent 按需 read 全文。
- **配置类走 skill + 热重载，而非专用工具**：避免堆窄工具；专用工具只给需要全自动流程的操作。
- **实例设置(GameSettings.Instance)不盲改**：有"覆盖标志 + 退出覆写"坑，edit_instance 只做 rename。
- **安全模式语义**：安全模式 = 仅危险操作二次确认，不是每次调用都确认。
- **子代理并行策略**：因 worktree 从会话起点旧提交分叉，只让子代理**新建文件**；改现有文件用主树内 `fork` 子代理或自己做。
- **诚实口径**：对外说"HMCL 之上约 1.7 万行 AI 系统"，而非"17 万行"。
- **测试策略**：CLI 全链路真模型测试优先于单元测试；用低智商 mercury-2 当基线。

## 4. 遇到的问题与解决

- **内置字体污染全局渲染**（两次）：导致中文/界面文本渲染异常 → 彻底撤销全局字体路线。
- **worktree 陈旧基线陷阱**：子代理从会话起点提交(`2233de23b`)分叉，看不到会话内提交，改现有文件的合并全部报废 → 放弃这些合并，只用新文件。写入长期记忆。
- **死文件误导子代理**：旧工具文件只取消注册未删除 → 删死文件。
- **"一坨屎"多参工具全失败**：plain Tool 只广播单个 `query`，模型把整个 JSON 塞进 query → 在适配器 `parseArguments` 通用解包（query 里的 JSON 自动拆成真实参数），一处修复所有工具。
- **install_loader 抛 UnsupportedOperationException**：调无参 `refreshAsync()` → 改按版本 `loadAsync/refreshAsync(gameVersion)`。
- **web_search 失败无错误回传**：WebSearchTool 已写好但未注册 → 注册 + execute 永不返回 null。
- **国内网络直连失败**：绕过了 HMCL 网络栈/镜像/代理 → 改走原生 DownloadProviders + `ProxySelector.getDefault()`。
- **ask 回车结束对话框**：发送按钮 `setDefaultButton(true)` → ask 时确认按钮设默认、发送取消默认。
- **PLAYBOOKS 死代码**：定义了但 build() 从未引用 → 接进 build()。
- **API 层面**：多次撞额度限流、上下文超 262144 token（触发 /compact）；deepseek-v4-pro[1m] 一度不可用切回 Opus 4.8。
